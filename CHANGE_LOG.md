## 2026-06-08

- SwiftUI "hello world" skeleton builds and runs on iPhone 15 Pro Max (Developer Mode enabled, device signed and trusted). V1.0 done.
- Project layout: `Relay Notes/Relay Notes/` holds source (`Relay_NotesApp.swift`, `ContentView.swift`, `Assets.xcassets`). Planning docs live at `Relay Notes/planning/` as a folder reference (not a target member).
- **Scope decision:** v1 is voice-to-text only. All LLM-driven cleanup, organization, and categorization moved to "Later." Doc restructured into v1 / Later sections.
- **V1.1 first pass shipped to device.** Mic → AVAudioRecorder (.m4a/AAC 64 kbps mono) → SpeechAnalyzer + SpeechTranscriber (file-based) → SwiftData. Functionally working end-to-end on iPhone 15 Pro Max. Accuracy on "some words" — tuning underway, see "V1.1 accuracy tuning" below.
- **Deviation from plan:** Used `AVAudioRecorder` instead of `AVAudioEngine` for V1.1. Reason: `AVAudioRecorder` writes AAC/m4a natively; `AVAudioEngine → file` requires `AVAudioConverter` middleware. `AVAudioEngine` returns in V1.2 when we need raw PCM buffers for streaming live transcription.
- **No tests yet:** No test target exists. Production code is structured to be testable (DI + protocols). Add a Unit Testing Bundle via Xcode (File → New → Target) before V1.2.
- **Tuning sheet shipped.** All four accuracy knobs are runtime-configurable from the in-app Tuning sheet (slider icon, top-right). `Tunings` (`@Observable`) holds state; `RecorderViewModel` reads a snapshot at the moment of recording/transcribing and passes `RecordingOptions` / `TranscriptionOptions` to the recorder/transcriber. Protocols updated to take options.
- **Tunings persistence shipped.** `Tunings` properties have `didSet` writes to `UserDefaults`; `init` reads back on launch. Settings survive cold starts.
- **V1.1 marked done.** Done-when criterion (speak → stored transcript offline) is satisfied. Accuracy tuning continues in parallel.
- **V1.2 step 1 shipped — notes list.** `ContentView` refactored to a `NavigationStack` with `NotesListView` on top (driven by `@Query` sorted by `createdAt` desc) and `RecorderView` at the bottom (Voice Memos pattern). Tapping a row navigates to a `NoteDetailView` placeholder. Tuning gear moved to the navigation bar.
- **V1.2 step 2 shipped — audio playback in `NoteDetailView`.** New `Audio/AudioPlayer.swift` is a `@MainActor @Observable` wrapper around `AVAudioPlayer`. Detail view shows: timestamp, selectable transcript, then a fixed bottom toolbar with a scrub slider, play/pause button, and elapsed/total time labels. Playback session uses `.playback` category. Player stops on `onDisappear`. Polling loop is a `Task` instead of `Timer` to keep Swift 6 strict concurrency clean.
- **V1.2 step 3 shipped — delete.** `Note.deleteWithAudio(in:)` removes the audio file from disk and the SwiftData row. Wired to swipe-to-delete on the list (no confirmation — swipe gesture is the confirmation) and a trash button in detail view (confirmation alert, since dismissing the screen is a less obvious "you're about to delete" cue). Mode default also changed: `.measurement` → `.default` based on real-world finding that AGC-disabled recordings are too quiet for comfortable playback.
- **V1.2 step 4 shipped — search.** `.searchable` on the navigation stack, in-memory case-insensitive filter on `note.transcript`. Empty-state uses `ContentUnavailableView.search(text:)` to distinguish "no matches" from "no notes." In-memory rather than SwiftData `#Predicate` because `localizedCaseInsensitiveContains` isn't reliably supported in `#Predicate` and at v1 dataset sizes the filter cost is negligible.

## 2026-06-09

- **Fixed — no transcription while screen locked (field bug).** Recording then locking the screen captured nothing: the live partial froze (showed `…` on unlock) and the locked window was missing from both transcript and saved audio. Root cause: the app declared no `audio` background mode, so iOS *suspended the process* on lock and the `AVAudioEngine` input tap stopped firing (it resumes on unlock — matching the reported symptom exactly). The `planning/notes.md` watch-item already flagged this ("Background recording … you need the background audio mode and a long-lived audio session"). Fix details worth knowing: `INFOPLIST_KEY_UIBackgroundModes` is a **no-op** — `UIBackgroundModes` isn't in Xcode's allowlist of generatable Info.plist keys (verified: the key never reached the built plist). So added a partial `Relay Notes/Info.plist` holding `UIBackgroundModes = [audio]` and set `INFOPLIST_FILE` in both build configs while keeping `GENERATE_INFOPLIST_FILE = YES` — Xcode merges the generated keys on top of that base file. The target uses a file-system-synchronized group, which auto-added the new `Info.plist` to Copy Bundle Resources → "Multiple commands produce Info.plist"; resolved with a `PBXFileSystemSynchronizedBuildFileExceptionSet` excluding `Info.plist` from membership. The audio session was already `.playAndRecord` and stays active for the whole recording, satisfying the "long-lived session" half. Verified: clean build, merged `Info.plist` contains `UIBackgroundModes=[audio]` *and* retains all generated keys (microphone/speech usage, orientations). **Needs on-phone confirmation** that locked-screen capture now works.
- **Known related gap (not fixed here).** No `AVAudioSession` interruption handling — a phone call, alarm, or another app grabbing audio mid-recording still stops `AVAudioEngine` with no auto-resume. Distinct from the suspension bug above; track as a follow-up if it bites in daily use.
- **Interruption handling shipped (closes the gap above).** `LiveAudioEngine` now observes `AVAudioSession.interruptionNotification` and emits an `AsyncStream<InterruptionEvent>` (`.began` / `.resumed` / `.stopped`) alongside the buffer stream. On `.began` the system has already stopped the engine; on `.ended` we honor the `shouldResume` hint — reactivate the session and `engine.start()` again (tap stays installed, so capture flows back into the *same* file and transcription session), or report `.stopped` when the hint says no or restart throws. `RecorderViewModel` gained a `.paused(partial:)` state: `.began` → paused (frozen partial), `.resumed` → recording, `.stopped` → auto-finalize so the captured audio isn't lost (no manual-resume affordance in the tap-to-record UI). `RecorderView` shows an orange "Paused — interrupted" status and keeps the partial card visible; the record button finalizes from `.paused` too. The interruption observer is delivered on `queue: .main` and bridged to the MainActor via `MainActor.assumeIsolated`; the observer/stream are torn down in `engine.stop()`.
- **Latent crash fixed in passing.** `LiveAudioEngine.stop()` now calls `removeTap(onBus:)` unconditionally instead of only when `engine.isRunning`. An interruption-stopped engine reports `isRunning == false` but still holds its tap, so the old guard would have left a stale tap and crashed the *next* `installTap`. `removeTap` is a no-op when no tap is installed, so unconditional removal is safe.
- **Still open (smaller).** Route changes (`AVAudioEngineConfigurationChange` / `routeChangeNotification` — e.g. unplugging headphones mid-record) aren't handled; the backgrounded-during-a-call edge needs on-device validation. Build is clean (Debug, no warnings); interruption behavior needs a real phone-call/Siri test to confirm.
- **First test target shipped — `Relay NotesTests` (Swift Testing).** The project's first unit-test target, hosted in the app (`@testable import Relay_Notes`). Covers the interruption → state mapping: `RecorderViewModel.handleInterruption` was refactored to delegate to a pure, static `nextState(for:from:)` so the transitions (`.began`→paused, `.resumed`→recording, and the guard/no-op cases) are testable without constructing the view model or touching hardware. 5 tests, all green via `xcodebuild test`. Added the target with the `xcodeproj` Ruby gem (not by hand — it round-trips the Xcode 16 synchronized groups safely) and validated the whole flow on a `/tmp` repo copy first. Scheme `Relay Notes` is now **shared** so the test action runs from the CLI. `CLAUDE.md` updated: testing how-to + the "edit the project via the gem, validate in /tmp" rule.

## 2026-06-10

- **T1 plan locked — Local Whisper via MLX, promoted ahead of the L stages.** Originally the Tier-2 transcription upgrade was "post-v1, when Apple's accuracy is the bottleneck *and* staying local matters" with no L-stage assigned. Decision today: do it next, before L1. Rationale: proving on-device third-party-model viability is the riskiest unknown of the local-first thesis, and the runtime we wire here is the same one L1+ LLMs will need on Apple silicon. New "Transcription upgrades" section in `planning/notes.md` between v1 and the L stages, with three items: **T1** (Local Whisper via MLX, next), **T2** (second on-device engine — Parakeet-MLX or Qwen3-ASR), **T3** (`CloudTranscriber`, opt-in). Tier-2 row of the transcription table relabeled model-agnostic ("Local ASR via MLX").
- **Picked raw `mlx-swift` over WhisperKit for T1.** Two reasons: (1) keeps the app to a single ML runtime — picking WhisperKit (Core ML under the hood) would split the runtime surface the moment L1's MLX LLMs land; (2) pays the MLX-on-iOS learning cost on a smaller problem (Whisper inference) than an LLM, with the lessons transferring directly to Parakeet-MLX / Qwen3-ASR / L1 model loading. Escape valve: drop to WhisperKit behind the same `Transcriber` protocol if `mlx-swift` proves intractable on iOS (~2 weeks of evening effort as the trigger). Documented in `planning/notes.md` (T1 risks watch-item) and `planning/transcription-tuning.md` (Decisions log).
- **T1 design decisions locked.** Default model: `mlx-community/whisper-small.en-mlx` (~250 MB, English-only). Stored in Application Support (not Caches — eviction risk), excluded from iCloud backup. Pre-download + delete affordances in Settings to preserve the offline-recording promise once installed. `TranscriptionOptions` becomes a sum type (`.apple` / `.whisperMLX`) so per-engine options stay type-safe; new `TranscriptionEngine` enum + `TranscriberFactory` resolves the engine per recording. First cut is **finalize-only** (no streaming partials) — recorder shows "Transcript will appear when you stop recording" placeholder + elapsed time + level meter while Whisper is the selected engine; `.finalizing` runs the full-file decode with a "Transcribing…" spinner. Buffer strategy: in-memory PCM during `feed`, decode once at `finish()` (~115 MB resident for 30 min of audio on iPhone 15 Pro Max — comfortable).
- **Issue #1 opened to track the in-memory buffer revisit trigger.** github.com/AlteredCraft/relay-notes/issues/1. Triggers to revisit: notes regularly exceed ~20 min, memory-pressure warnings under dogfood, or device scope expands beyond the iPhone 15 Pro Max. Options at revisit time documented (scratch WAV during `feed`, reuse the AAC writer's file, or chunked decode).
